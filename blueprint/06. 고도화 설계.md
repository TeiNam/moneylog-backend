# 🔧 고도화 설계 (데이터 무결성·AI·UX)

## 1. 계좌 간 이체 (내부 이체)

### 1-1. 문제

현재 단식 부기 구조에서는 "통장 → 신용카드 대금 납부"를 표현할 수 없음.
카드 대금 결제, 통장 간 이체, 현금 인출 등은 수입도 지출도 아닌 자산 이동.

### 1-2. 내부 이체 테이블 (transfer)

| 필드 | 타입 | 설명 |
|------|------|------|
| id | UUID | PK |
| user_id | UUID | FK → user (실행자) |
| family_group_id | UUID | nullable |
| from_asset_id | UUID | FK → asset (출금 자산) |
| to_asset_id | UUID | FK → asset (입금 자산) |
| amount | BIGINT | 이체 금액 |
| date | DATE | 이체일 |
| description | VARCHAR | 내용 (예: "3월 삼성카드 대금") |
| transfer_type | VARCHAR | CARD_PAYMENT / ACCOUNT_TRANSFER / CASH_WITHDRAWAL / CASH_DEPOSIT / OTHER |
| memo | TEXT | 메모 |
| created_at | TIMESTAMPTZ | 생성일시 |

**transfer_type 상세:**

| 유형 | 예시 | from → to |
|------|------|-----------|
| CARD_PAYMENT | 카드 대금 납부 | 통장 → 신용카드 |
| ACCOUNT_TRANSFER | 통장 간 이체 | 급여통장 → 저축통장 |
| CASH_WITHDRAWAL | 현금 인출 | 통장 → 현금 |
| CASH_DEPOSIT | 현금 입금 | 현금 → 통장 |
| OTHER | 기타 | 자유 |

### 1-3. 이체 입력 화면

```
┌──────────────────────────────┐
│  🔄 이체                      │
├──────────────────────────────┤
│  유형: [카드 대금 납부 ▼]     │
│                              │
│  출금: [🏦 국민 급여통장 ▼]  │
│  입금: [💳 삼성 신용카드 ▼]  │
│  금액: ₩ [820,000]          │
│  날짜: [2026-03-25]         │
│  내용: [3월 삼성카드 대금]    │
│                              │
│  [이체 실행]         [취소]  │
└──────────────────────────────┘
```

### 1-4. 이체가 자산 잔액에 미치는 영향

- 이체는 transaction에 기록되지 않음 (수입/지출이 아님)
- 자산 잔액 계산 시 transfer도 반영:
  - `from_asset`: 잔액 감소
  - `to_asset`: 잔액 증가 (카드의 경우 사용액 차감)
- 통계/지출 분석에는 포함되지 않음

## 2. 구독 결제 자동 생성 (배치)

### 2-1. 동작 흐름

```
[EventBridge 스케줄러] ── 매일 00:05 KST ──→ [배치 API 호출]
                                                │
                                                ▼
                                    [오늘 결제일인 구독 조회]
                                    SELECT * FROM subscription
                                    WHERE status = 'ACTIVE'
                                      AND billing_day = today
                                                │
                                                ▼
                                    [transaction 자동 생성]
                                    source = 'SUBSCRIPTION_AUTO'
                                                │
                                                ▼
                                    [푸시 알림 발송]
                                    "넷플릭스 17,000원 결제됨"
```

### 2-2. 배치 로직 상세

```python
# app/services/subscription_batch.py (핵심 로직)

async def process_daily_subscriptions(db: AsyncSession):
    today = date.today()

    # 오늘 결제일인 활성 구독 조회
    subs = await db.execute(
        select(Subscription)
        .where(Subscription.status == "ACTIVE")
        .where(Subscription.billing_day == today.day)
    )

    for sub in subs.scalars():
        # 중복 방지: 이번달 이미 생성된 건 스킵
        exists = await db.execute(
            select(Transaction)
            .where(Transaction.source == "SUBSCRIPTION_AUTO")
            .where(Transaction.description == sub.service_name)
            .where(Transaction.date == today)
        )
        if exists.scalar():
            continue

        # transaction 자동 생성
        tx = Transaction(
            user_id=sub.user_id,
            family_group_id=sub.family_group_id,
            date=today,
            area="SUBSCRIPTION",
            type="EXPENSE",
            major_category="구독",
            minor_category=sub.category,
            description=sub.service_name,
            amount=sub.amount,
            actual_amount=sub.amount,
            asset_id=sub.asset_id,
            source="SUBSCRIPTION_AUTO",
        )
        db.add(tx)

    await db.commit()
```

### 2-3. 엣지 케이스 처리

| 케이스 | 처리 |
|--------|------|
| 결제일이 31일인데 2월 | 해당 월 마지막 날로 대체 |
| 연간 구독 (YEARLY) | billing_day + 가입월 기준으로 연 1회만 생성 |
| 주간 구독 (WEEKLY) | 매주 해당 요일에 생성 |
| 구독 일시중지 (PAUSED) | 배치에서 제외 |
| 서버 장애로 배치 미실행 | 다음날 배치 시 전날 누락분 보정 |

### 2-4. 결제 전 알림

```
[EventBridge] ── 매일 09:00 KST ──→ [알림 배치]
                                       │
                                       ▼
                            [N일 후 결제 예정 구독 조회]
                            WHERE billing_day = today + notify_before_days
                                       │
                                       ▼
                            [푸시 알림]
                            "내일 넷플릭스 17,000원 결제 예정"
```

## 3. AI 고도화

### 3-1. 영수증 OCR (멀티모달)

```
[사용자: 영수증 사진 촬영/업로드]
         │
         ▼
[S3 업로드] → [Bedrock Claude (Vision)]
                │
                ├─ 추출 항목:
                │   - 상호명
                │   - 결제일시
                │   - 항목별 금액
                │   - 총 금액
                │   - 결제수단 (카드 끝자리)
                │
                ▼
         [구조화된 JSON]
         {
           "store": "이마트 성수점",
           "date": "2026-03-09",
           "items": [
             {"name": "우유", "amount": 3200},
             {"name": "계란", "amount": 6500}
           ],
           "total": 9700,
           "card_last4": "1234"
         }
                │
                ▼
         [자동 매칭]
         - card_last4 → 사용자 카드 자동 매칭
         - store명 → 카테고리 자동 분류
                │
                ▼
         [확인 화면 → 저장]
```

### 3-2. 프롬프트 컨텍스트 설계

AI 채팅 호출 시 시스템 프롬프트에 사용자 컨텍스트 주입:

```python
def build_system_prompt(user, assets, categories):
    return f"""당신은 가계부 입력 도우미입니다.

## 사용자 정보
- 닉네임: {user.nickname}

## 보유 자산 (결제수단)

2. 결제수단이 언급되면 위 자산 목록에서 매칭하세요
3. "주유", "톨비" 등 → area: CAR
4. "넷플릭스", "유튜브" 등 → area: SUBSCRIPTION

```

### 3-3. AI 오분류 학습

- 사용자가 AI 제안을 수정한 경우 → 수정 이력 저장
- 같은 패턴 반복 시 → 사용자 선호 학습 (예: "스타벅스" → 항상 "커피숍"으로 분류)
- 학습 데이터는 프롬프트 컨텍스트에 추가:

```
## 사용자 선호 (과거 수정 이력)
- "스타벅스" → 식비 > 커피숍
- "쿠팡" → 생활용품비 > 기타생활용품
- "주유소" → 차계부 (area: CAR)
```

## 4. 비밀 모드 (Private Tag)

### 4-1. 개념

가족 공유 모드에서 특정 거래를 다른 구성원에게 숨기는 기능.

### 4-2. 구현

`transaction` 테이블에 필드 추가:

| 필드 | 타입 | 설명 |
|------|------|------|
| is_private | BOOLEAN | 비밀 거래 여부 (기본 false) |

**동작:**
- `is_private = true`인 거래는 본인에게만 표시
- 가족 통합 뷰에서 제외
- 통계에서도 가족 합산 시 제외 (본인 개인 통계에만 포함)
- 카드 정산 시: 비밀 거래 금액은 정산 대상에서 제외 (본인 부담)

### 4-3. 입력 시 UI

```
┌──────────────────────────────┐
│  지출 입력                    │
├──────────────────────────────┤
│  금액: ₩ [50,000]           │
│  ...                         │
│                              │
│  🔒 비밀 거래  [OFF ▶ ON]   │
│  ⓘ 가족 구성원에게 보이지    │
│    않으며, 정산에서 제외됩니다│
│                              │
│  [저장]              [취소]  │
└──────────────────────────────┘
```

### 4-4. 가족 뷰에서의 표시

```
가족 통합 뷰 (아내가 보는 화면):
  03/09 👤남편 점심      -12,000  ← 보임
  03/09 👤남편 주유      -82,000  ← 보임
  03/09 👤남편 ???       (숨겨짐)  ← 비밀 거래, 아예 안 보임

남편 본인 뷰:
  03/09 점심             -12,000
  03/09 주유             -82,000
  03/09 🔒 선물          -50,000  ← 본인만 보임
```

## 5. 오프라인 모드 & 동기화

### 5-1. 전략

```
[오프라인 입력] → [로컬 DB (SwiftData)] → [네트워크 복구] → [서버 동기화]
```

### 5-2. 동기화 방식

| 항목 | 방식 |
|------|------|
| 로컬 저장 | SwiftData (iOS) / SQLite (웹 IndexedDB) |
| 동기화 트리거 | 네트워크 복구 시 자동 + 앱 포그라운드 진입 시 |
| 충돌 해결 | Last-Write-Wins (updated_at 기준) |
| 동기화 단위 | 변경된 레코드만 (delta sync) |
| 동기화 상태 | 각 레코드에 sync_status 필드 |

### 5-3. 로컬 레코드 sync_status

| 상태 | 의미 |
|------|------|
| SYNCED | 서버와 동기화 완료 |
| PENDING_CREATE | 로컬에서 생성, 서버 미반영 |
| PENDING_UPDATE | 로컬에서 수정, 서버 미반영 |
| PENDING_DELETE | 로컬에서 삭제, 서버 미반영 |
| CONFLICT | 서버와 충돌 (수동 해결 필요) |

### 5-4. 동기화 흐름

```
[앱 시작 / 네트워크 복구]
         │
         ▼
[PENDING 상태 레코드 조회]
         │
         ▼
[서버 API 일괄 전송]
POST /api/sync
{
  "creates": [...],
  "updates": [...],
  "deletes": [...]
}
         │
         ▼
[서버 응답: 성공/충돌 목록]
         │
         ├─ 성공 → sync_status = SYNCED
         └─ 충돌 → sync_status = CONFLICT
                   → 사용자에게 해결 UI 표시
```

### 5-5. 오프라인 시 UI 표시

```
┌──────────────────────────────┐
│  ⚡ 오프라인 모드              │
│  입력한 내역은 네트워크 복구  │
│  시 자동으로 동기화됩니다     │
├──────────────────────────────┤
│  03/09 점심 김치찌개 -12,000 │
│       ☁️ 동기화 대기 중       │
│  03/09 커피        -4,500   │
│       ☁️ 동기화 대기 중       │
└──────────────────────────────┘
```

## 6. 기존 테이블 변경 요약

| 변경 | 내용 |
|------|------|
| 신규 테이블 | `transfer` (계좌 간 이체) |
| `transaction` 필드 추가 | `is_private` (BOOLEAN, 기본 false) |
| API 신규 | `transfers.py` (이체 CRUD) |
| API 신규 | `sync.py` (오프라인 동기화) |
| 배치 신규 | 구독 자동 결제 생성 (EventBridge + API) |
| 배치 신규 | 결제 전 알림 발송 (EventBridge + SNS) |
